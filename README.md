# Adaptive RAG Data Platform

A full-stack Retrieval-Augmented Generation (RAG) system with **LLM-driven dynamic UI generation** — the interface adapts its layout to match the content type of each query response. Built for EE 547.

---

## What makes it different

Most RAG systems return plain text. This one returns a structured UI schema alongside every answer. The frontend validates the schema and renders the appropriate component — a table, chart, timeline, image viewer, audio player, or code editor — without any hardcoded routing logic.

- **Dynamic layouts:** each query response carries a UI schema (JSON) generated by content-type heuristics locally, or by Bedrock LLM on AWS
- **12 leaf components:** `text`, `kpi`, `table`, `chart`, `card`, `list`, `timeline`, `form`, `source_refs`, `image_viewer`, `code_editor`, `audio_player`
- **UI cache:** repeated query patterns reuse cached schemas; a `Cached layout` badge appears in the UI — layout always rebuilds fresh from current sources on a hit
- **Role-based access control:** admin, contributor, and user roles with per-file ownership and visibility

## Architecture

```
Browser (React + Vite)
    │
    │  REST/JSON
    ▼
FastAPI (Python) on EC2
    ├── /auth        JWT login, user management
    ├── /ingest      File, JSON, URL ingestion
    ├── /jobs        Job status, file serving, visibility
    ├── /query       Retrieval + answer + UI schema
    ├── /records     Raw record access
    └── /ui-schema   Schema cache inspection
    │
    ├── Chunking pipeline
    │     detect → chunk → embed → index → persist
    │
    ├── Retrieval
    │     SentenceTransformer embeddings + cosine similarity (local)
    │     OpenSearch Serverless k-NN (AWS)
    │
    ├── Answer generation
    │     Ollama llama3.2 (local) or Bedrock Llama 3.3 70B (AWS)
    │
    └── UI schema generation
          Heuristic keyword routing (local) or Bedrock LLM (AWS)
```

**Local mode** runs entirely on your machine — no AWS account needed. **Cloud mode** swaps in DynamoDB, S3, OpenSearch Serverless, and Bedrock.

## Supported file types

| Type | Extensions | Rendered as |
|---|---|---|
| Images | `.jpg` `.jpeg` `.png` `.gif` `.webp` `.bmp` `.tiff` | Image Viewer |
| Code | `.py` `.js` `.ts` `.tsx` `.go` `.rs` `.java` `.cpp` `.c` `.sql` `.yaml` and more | Code Editor with syntax highlighting |
| Audio | `.mp3` `.wav` `.flac` `.ogg` `.m4a` `.aac` `.wma` | Audio Player |
| PDF | `.pdf` | Extracted and indexed as prose |
| Tabular | `.csv` | Table / KPI layout |
| JSON | `.json` or paste | Chart / KPI / table |
| Prose | `.txt` `.md` | Text + source refs |

Images are described at ingest time by Ollama `llava` (local) or Bedrock Llama 4 Maverick (AWS) so they are fully queryable. Code files are chunked and concatenated in document order by the viewer.

## Roles

| Action | Admin | Contributor | User |
|---|---|---|---|
| Upload files | ✅ | ✅ | ❌ |
| See Files tab | ✅ all files | ✅ own files | ❌ |
| Delete files | ✅ any | ✅ own only | ❌ |
| Toggle visibility | ✅ | ❌ | ❌ |
| Query | ✅ all data | ✅ own + public | ✅ public only |
| Manage users | ✅ | ❌ | ❌ |

Files default to **private**. Admins can mark any file public to make it queryable by all roles.

## Running locally

**One command:**

```bash
./scripts/run_local.sh
```

This starts Ollama, pulls models if missing, waits for the backend to be ready, and starts the frontend. Logs go to `logs/backend.log` and `logs/frontend.log`. Press `Ctrl+C` to stop everything.

**Manual setup:**

```bash
# 1. Install dependencies
pip install -r backend/requirements.txt
cd frontend && npm install

# 2. Pull Ollama models
ollama pull llama3.2   # text answers
ollama pull llava      # image description (optional)

# 3. Start everything (3 terminals)
ollama serve
USE_LOCAL_STORE=true EMBEDDING_PROVIDER=local uvicorn backend.api.main:app --reload --port 8000
cd frontend && npm run dev
```

Open [http://localhost:5173](http://localhost:5173). Default login: `admin` / `changeme`.

**Seed demo data:**

```bash
python scripts/seed_demo.py
```

Loads 5 datasets (CSV, JSON, TXT) and polls until all are `READY`.

## Tech stack

**Backend**
- FastAPI + Uvicorn
- SentenceTransformers (`all-MiniLM-L6-v2`) for local embeddings
- Ollama (`llama3.2` answers, `llava` vision) for local LLM
- AWS Bedrock (`Llama 3.3 70B` answers, `Llama 4 Maverick` vision, `Titan` embeddings) for cloud
- DynamoDB / S3 / OpenSearch Serverless on AWS; local JSON stores for dev
- JWT auth with `python-jose` + `passlib[bcrypt]`

**Frontend**
- React 18 + TypeScript + Vite
- Tailwind CSS with dark mode
- TanStack Query for server state
- AJV for UI schema validation
- `react-syntax-highlighter` for code, `recharts` for charts

**Infrastructure**
- AWS CDK (TypeScript)
- EC2 + nginx for backend and frontend hosting
- OpenSearch Serverless with HNSW k-NN index (faiss engine)
- GitHub Actions CI/CD — push to `main` auto-deploys via AWS SSM (no open ports required)

---

## CI/CD

Pushing to `main` triggers a GitHub Actions workflow that deploys to both EC2 instances via AWS SSM (no SSH port needed). The frontend public IP is posted to the Actions run summary after each deploy.

Required GitHub Secrets: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`, `BACKEND_INSTANCE_ID`, `FRONTEND_INSTANCE_ID`.
